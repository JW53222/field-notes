# The duplicate detector lives in the write loop

*Published July 12, 2026. The events: April 21–26, 2026 (the standing sweep) and June 2 (the merge-gate stage), with the origin story reaching back to March. Dates, commit IDs, and quoted messages come from the git history. Tweaks have landed since; the commits cited are the ones that built the thing.*

Everyone puts embeddings in front of the model: semantic search as retrieval, so the agent can find code before it writes. I ended up also pointing them *backwards* — at the model's own output, inside the merge gate — because of a problem every multi-agent codebase eventually meets: **agents reinvent your helpers.** Not copy them — reinvent them, slightly differently, under a new name, in a plausible location. Each duplicate is invisible to grep, splits future callers into camps, and doubles the surface every later agent has to read. And reading is the expensive operation: the whole reason I had embeddings at all is that it cost too much to have Claude re-read the codebase to find things out.

## Built in a day, verified for a week

The harness landed on April 21 in a single day's work (`b213bc528`): executable invariants, structural detectors, **semantic detectors** — embedding-based checks that compare what an agent just wrote against what already exists — and drift tracking, together as one audit pipeline. The next day's commit is the origin story writing itself: "Rebuild audit-hardening work lost after session…" — the harness's first production act was recovering its own lost construction.

Then the tuning, which is where the honesty lives. April 24: the embedding cache moved to float16 (“8x smaller, ~20x faster”), because a detector that's too slow to run on every merge is a detector that doesn't run. April 26: five triage waves in one day, each one adversarially reviewed, and the commit messages report their own null results — "triage wave-11 — 3-lane detector improvements, **0 real bugs**" (`7c271e422`), "wave-13 — incorporate deferred oppositional findings" (`77d64bc2f`). A detector suite whose changelog admits *this wave found nothing real* is a detector suite you can calibrate. One whose every wave claims victory is a slot machine.

## What it actually does, and when

Two detectors bracket the edit, and the receipts demand their dates be kept apart.

**April 21: the standing sweep (the just-after side).** The audit harness's semantic detectors run over the *whole* corpus, not a diff: function bodies normalized (locals renamed, comments stripped), embedded, density-clustered to catch paraphrase twins across modules, and the clusters joined with the callgraph overlay so they're sorted by suspicion before triage. This is consolidation — find every pair of functions that are the same idea wearing two names, wherever and whenever they landed.

**June 2: the gate advisory (the just-before side).** A dedicated stage inside the merge gate embeds a branch's new and modified functions and compares them against the existing corpus. The threshold is honest about itself: 0.78 cosine, set from a hand-labeled sweep where real reinventions scored 0.82–0.89 and genuinely novel code stayed at or below 0.72, and documented in-code as a provisional, warn-first default. A hit isn't auto-rejected — similarity is a lead, not a verdict — it's reported with the existing twin named, and the adjudication step decides: justify the second copy or call the first one.

The effect compounds: every duplicate that doesn't land is context the next hundred agents don't have to read, which in an agent codebase is the scarcest resource there is.

The same vectors turned out to be the seed of something bigger. Deduplication was the first consumer; then the corpus got merged with the dependency map and injected into agents' context at file-touch time, and by the time I learned the industry sells that combination as "hybrid retrieval," I'd been running it for weeks (the [longer story](2026-07-12-from-zero-with-receipts.md) has that lineage, with the numbers).

## The claim, corrected twice in one day

The first version of this post ended with "nothing I've found audits what the model writes against what it already wrote." I sent an adversarial research agent to refute that, corrected the post on its findings — and then, suspecting the correction had overcorrected, sent a second agent to audit the first. Both passes changed the text. Here's where it landed.

**The concept is prior art.** [Greptile](https://www.greptile.com) — a shipped, established review product — indexes your whole repo, and its PR comments reference similar code that already exists specifically to prevent repetition. That's semantic comparison of new code against the existing tree, surfacing the twin, in production before my April build. [Octopus Review](https://octopus-review.ai/blog/your-ai-writes-the-same-code-5-times) (open-source, first release March 24, 2026) embeds the codebase into a vector store for context-aware review, and its blog describes the exact agents-reinvent-your-helpers problem — though the dedicated dup-detection framing lives in its marketing, not its documented feature list. Either way: comparing newly written code against a semantic index and pointing at the prior implementation was not my invention.

**The first correction mis-cited [Pharaoh](https://pharaoh.so/blog/prevent-duplicate-functions-ai-coding/).** The second agent read it properly: Pharaoh is a read-side structural knowledge graph — tree-sitter into Neo4j, signatures and call edges, no embeddings — that the agent *queries before it writes*. It's a good tool, and it's prior art for asking-before. But it does neither of the two things this loop does. It doesn't *push* context into the agent unasked — it waits to be queried, which is a lookup, not injection. And it doesn't *verify uniqueness after* the write — it's a pre-write search, prevention, where this post is about post-merge detection. Pre-write pull versus post-write check: opposite side of the edit, and the half Pharaoh has isn't the half this post claims. (I caught the mis-cite myself on a re-read; the second agent confirmed it. The review process reviews the reviewers — apparently just how everything works here now.)

**What survived both passes: the locus.** Every tool either agent could document is an external bot that comments on an opened PR for human readers, or a graph the model consults before writing. Mine is a stage *of the pipeline that owns the merge* — the gate advisory and the standing sweep bracketing the edit itself, the twin handed to an adjudication step, in a loop where the authors being audited are agents. To be precise about what's *not* the claim: pre-merge timing alone isn't novel — a CLI in CI runs pre-merge too. The parts no shipped tool demonstrated are the bracket as a whole, and especially the just-after side: a periodic whole-tree consolidation sweep joined to the callgraph, which diff-scoped PR bots don't do. The second reviewing agent also read the implementation itself and confirmed this description against the code, file by file. That integration is the only claim still standing, and the only one I'm making.

The residue I'll own instead of a trophy: seventeen weeks into software, I converged on what turned out to be a live product category in five days, from token economics and lived annoyance — and when I overclaimed past that, my own review process caught it. Twice. That second sentence is the skill this post is actually about.
